TIER 1Codzienne, używasz prawie zawsze

Plan mode

Naciskasz Shift+Tab, żeby przełączać tryby. Plan mode = ja planuję, nic nie wykonuję, dopóki nie zatwierdzisz. Ratuje przed sytuacją „Claude coś zrobił, a ja nie chciałem”. Domyślnie używaj dla zadań, które dotyczą >2 plików.

Sub-agenci — sekret oszczędzania tokenów

To są „moje pomocnicze instancje” w odrębnym oknie kontekstu. Gdy wysyłam zadanie do sub-agenta, dostaję od niego tylko podsumowanie, nie cały bałagan z eksploracji.

AgentDo czegoPrzykład
ExploreSzybkie szukanie w kodzie„Znajdź wszystkie miejsca, gdzie używamy AnthropicAPI”
PlanProjektowanie implementacji„Zaprojektuj dekompozycję BaselinkerWorkflowPanel”
general-purposeWieloetapowe taski research„Sprawdź jak działa tłumaczenie i jakie są bottlenecks”

Realny zysk: gdy Explore agent przeszuka 50 plików, zwraca mi 1 raport w 200 słów zamiast 50 plików w moim kontekście. Ja zostaję czysty, Ty zyskujesz miejsce na faktyczną pracę.

Wniosek: zamiast prosić mnie „rozejrzyj się po kodzie” — proś „wyślij Explore agenta, żeby się rozejrzał po kodzie”. To ja decyduję, ale możesz mi przypomnieć.

MCP — Model Context Protocol (rozszerzenia dla Claude)

Programy zewnętrzne, które dają mi nowe narzędzia. Np.:

  • GitHub MCP → mogę otwierać PR-y, czytać issues, robić komentarze
  • Playwright MCP → mogę debugować scraper przez prawdziwą przeglądarkę
  • SQLite MCP → mogę odpalić query do db.ts i sprawdzić plan zapytania
  • Obsidian MCP → czytam/piszę do Twojego vaulta

Jak wyglądają w praktyce: dodajesz wpis w .claude/settings.json (mcpServers) → restart Claude Code → masz nowe narzędzia.

Koszt kontekstu: każdy MCP dodaje ~100–500 tokenów (opisy narzędzi). Dlatego NIE instaluj 20 MCP „na zapas”. Tylko te, których faktycznie używasz.

Już masz włączone przez Anthropic: Google Drive, Gmail, Calendar, Canva — widać w settings.

TIER 3Zaawansowane (na potem, jak będziesz miał potrzebę)

Worktree / paralelne sesje

Mogę otworzyć drugą kopię projektu w branchu i pracować równolegle. Przydatne, gdy: „zacznij refaktor X w tle, ja nadal robię Y w głównej sesji”. Komenda EnterWorktree. Skomplikowane, na razie pomiń.

Custom skills (Twoje własne komendy)

Tworzysz plik .claude/skills/twoja-komenda.md → wpisujesz /twoja-komenda w czacie i to się odpala. Np. mogę Ci zrobić /check-baselinker-route, która ma instrukcje, jak konkretnie sprawdzać route'y do BaseLinkera. Sens dopiero, gdy powtarzasz tę samą instrukcję często.

Custom sub-agenci

Plik .claude/agents/twoj-agent.md → masz własny typ agenta (oprócz Explore/Plan). Np. agent-baselinker-expert, który zna tylko Twój kod BL. Sens dopiero, gdy projekt mocno wyspecjalizowany.

Background tasks i CronCreate

Mogę uruchomić zadanie w tle (run_in_background: true) lub na cron'ie. Np. „raz dziennie sprawdź czy są nowe issue na GitHubie i podsumuj”. Sens dla automatyzacji.

Output styles

/effort low = wyłączam extended thinking dla prostych zadań (oszczędność). Domyślnie sam decyduję, ale możesz wymusić.

Prompt caching (jest automatyczne, nie musisz nic robić)

Anthropic cache'uje moje prompty na 5 minut. Stąd: szybciej i taniej, gdy wracasz z follow-up pytaniem w ciągu 5 minut. Po 5 min cache wygasa → następna wiadomość jest droższa (recachowanie). Praktycznie: nie zostawiaj sesji „na potem za godzinę”, jeśli zaraz wracasz.

Czego NIE musisz wiedzieć (mit / overrated)

  • SuperClaude / BMAD / claude-flow / awesome-claude-code-toolkit „135 agentów” — nadal mówię to samo: noise. Wbudowane skills + sub-agenci + 2–3 MCP załatwiają wszystko bez konfliktu.
  • „Token-saving frameworks” — auto-compaction + sub-agenci + prompt caching są wbudowane i lepsze niż zewnętrzne hacki.
  • Większość „agent harness” repo z GitHuba — duplikują wbudowane funkcje i konfliktują z nowszymi wersjami Claude Code.

Praktyczna checklist „co odpalić w nadchodzącym tygodniu”

  1. Otwórz CLAUDE.md — przeczytaj, popraw, jeśli coś źle
  2. Zacommituj CLAUDE.md + .claude/ razem z SETUP.md / ENV.md / DEPENDENCIES.md
  3. Spróbuj /cost — zobacz, ile do tej pory wydałeś
  4. Spróbuj /context — zobacz, ile tokenów teraz zjadłeś w naszej sesji
  5. W następnym zadaniu użyj Plan mode (Shift+Tab) — zobacz różnicę
  6. Przed następnym commitem do API odpal /security-review
  7. Zainstaluj ccusage (npm i -g ccusage → ccusage) — zobacz dzienne wydatki

Po tym tygodniu będziesz wiedział, co Ci faktycznie pasuje i o co dopytać dalej. Reszta (MCP, custom skills, Obsidian) poczeka, aż będziesz miał konkretny problem do rozwiązania.

Sekcje settings.json oprócz hooks

Wszystko jest opcjonalne. Najczęstsze:

{
  "hooks": { ... },           // już znasz

  "mcpServers": {              // MCP servers
    "playwright": { ... },
    "sqlite": { ... }
  },

  "permissions": {             // co wolno bez pytania
    "allow": ["Bash(npm run lint)", "Bash(git status)"],
    "deny": ["Bash(rm -rf*)", "Write(.env*)"]
  },

  "env": {                     // zmienne dla wszystkich Bash
    "NODE_OPTIONS": "--max-old-space-size=4096"
  },

  "model": "claude-opus-4-7",  // wymuś konkretny model dla tego projektu

  "outputStyle": "concise",    // styl mojego pisania

  "statusLine": { ... },       // niestandardowy pasek statusu

  "ignorePatterns": [          // pliki które nigdy nie czytam (oprócz .gitignore)
    "*.log", "dist/**"
  ],

  "autoCompactThreshold": 80   // procent kontekstu kiedy uruchomić /compact auto
}

Najbardziej przydatne dla Ciebie z tej listy (po hooks):

  1. permissions.allow — nie będę pytał za każdym razem o npm run lint. Oszczędza klikania.
  2. permissions.deny — np. zabronisz mi pisać do .env*, żeby przypadkiem nie nadpisał.
  3. mcpServers — gdy zdecydujesz na MCP (Playwright, SQLite, GitHub).

Przykład sensownego permissions dla Twojego projektu:

{
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run build)",
      "Bash(git status)",
      "Bash(git diff*)",
      "Bash(git log*)"
    ],
    "deny": [
      "Write(.env)",
      "Write(.env.local)",
      "Write(google-credentials.json)",
      "Bash(rm -rf*)"
    ]
  }
}

Z tym: nigdy nie nadpiszę przypadkiem sekretów + nie pytam o read-only komendy gita.

Co możesz dodać teraz, gdy zechcesz

  1. ~/.claude/CLAUDE.md (globalny) — preferencje dla wszystkich Twoich projektów. Np.: „pisz po polsku, nie używaj emoji, preferuj minimalizm”. Nie powtarzasz się w każdym projekcie.
  2. permissions w .claude/settings.json — przykład wyżej. Mniej klikania + ochrona sekretów.
  3. .claude/skills/ — gdy zauważysz, że często prosisz mnie o to samo (np. „sprawdź wydajność tego SQL query”), zamień na slash command /check-sql.
  4. .claude/agents/ — gdy zauważysz, że często prosisz Explore agenta z TYM samym briefem (np. „rozejrzyj się po kodzie BaseLinker”), zamień na własnego agenta.

Czego prawdopodobnie NIE potrzebujesz

  • output-styles/ — defaulty są OK, custom dopiero, jak będziesz miał konkretną pretensję do mojego pisania
  • keybindings.json — domyślne wystarczają w 99% przypadków
  • Subdirectory CLAUDE.md (np. src/lib/CLAUDE.md) — Twój projekt nie jest aż tak duży

Kolejność, jeśli chcesz rozbudowywać

  1. Najpierw permissions w istniejącym .claude/settings.json — natychmiastowy efekt (mniej klikania) za 30 sekund roboty
  2. Potem globalny ~/.claude/CLAUDE.md z Twoimi preferencjami uniwersalnymi (po polsku, minimalizm, szczerość) — wpływa na wszystkie sesje
  3. Dopiero potem custom skills/agents — gdy zauważysz powtarzające się prośby
